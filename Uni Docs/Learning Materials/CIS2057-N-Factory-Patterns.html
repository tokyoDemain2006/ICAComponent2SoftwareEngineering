The Simple Factory Pattern

What’s the problem with new anyway?

  • Imagine you are writing code for a MMORPG which works out the type of item which is dropped when an enemy dies.
  • There are three types of item, all sub-classes of Item:
    • CommonItem
    • RareItem
    • EpicItem
  • The type of item the user actually gets is related to several factors, which are only known at run-time e.g.
    • The level of the user's character.
    • The number and level of party members.
    • The level of the enemy character.
    • Environmental factors such as game time.

The simple factory

But a high-level character that defeats a minor creature might only get a common item.

DRY – Don’t Repeat Yourself.

You re-factor the Item creation code into a method that can be re-used.

  • Then your Boss tells you that he wants to enrich the scenario in the game i.e. make regular changes to your code!
    • For example, he wants to add to the range of items returned by the method e.g. VeryCommonItem.
  • You realise that the createItem() method is going to change a lot now, and a lot in the future – DESIGN PRINCIPLE VIOLATION!
  • with the Strategy pattern, you need some way of encapsulating this change and removing it from your core code:
    • The Strategy pattern encapsulated a family of algorithms.
    • What you need is a way to encapsulate the instantiation of objects.

Things change

Encapsulate what varies. In other words, separate the parts of your code that will change the most from the rest of your application, and re-use where possible.

  • looks like we’re trying to avoid the use of new!
  • But what’s wrong with new?
  • Nothing, however whenever we use new, we are instantiating a concrete class, i.e. an implementation, not an interface.
  • And yet we have seen that programming to an implementation renders your code more fragile and error-prone. Why?
  • this example, instantiation takes place at run-time depending on some sort of condition (difficulty level).
  • Code like this violates the Open-Closed Principle.

The simple factory

  • What have we done so far to avoid programming to an implementation?
  • have programmed to a super-type (e.g. an interface).
  • coding to an interface, you can insulate yourself from a lot of changes that might happen to a system in the future. Why?
  • Because if your code is written to an interface, then it will work with any new classes implementing that interface through polymorphism.

The simple factory

  • there is nothing wrong with using new.
    • The real problem is change & the way change impacts our use of new.
    • When you have code that chooses between a range of objects based on run-time conditions you are heading for trouble because:
      • That code may have to be changed as new concrete classes are added to the range.
      • the criteria for object selection changes.
    • The simple factory decouples the client from this business of object creation.

The Factory Method Pattern

Let’s take it one step further.

  • The simple factory is used extensively in Software Engineering.
  • you ask most programmers to define the factory pattern, they will start talking about simple factories.
    • But there is an important difference between the simple factory and the GoF factory method pattern.
  • defined by the gang of four (GoF), the Factory Method pattern is something more flexible.
    • According to their definition, the factory method “defines an interface for creating an object, but let subclasses decide which class to instantiate.”
    • Factory Method lets a class defer instantiation to subclasses.

The Factory Method Pattern

  • The key here is in the line ‘let the subclasses decide’.
  • far, we have not yet considered extending the factory class.
  • But what if we encounter a situation where a simple factory class is no longer appropriate for all the clients.
    • For example, the player of our MMORPG enters a realm where the item rewards are different:
      • ElvishCommonItem.
    • the player passes some level threshold that changes the type of items that should be awarded:
      • UltimateItem.

The Factory Method Pattern

  • this case, we need to modify the factory class, which we can do by simply extending it:
    • ElvenItemFactory():IItemFactory
  • But it would be much better to define an interface, which can be implemented as necessary.
    • This gives us more flexibility.

Abstract Factory Pattern

Just a little deeper into the rabbit hole…

A definition

The book Design Patterns: Elements of Reusable Object-Oriented Software states that an Abstract Factory provides an interface for creating families of related or dependent objects without specifying their concrete classes

The client software creates a concrete implementation of the abstract factory, and then uses the generic interface of the factory to create the concrete objects that are part of the theme.

It can help to think of an abstract factory as a “super-factory”.

Let’s take another game example

Destiny (by Bungie) offers different rewards drops not to dissimilar to our quest/item examples from earlier. These can be influenced by “events” through the games lifecycle that are usually tied to seasons.

These can be one offs (The Dawning in Christmas 2017), or regular (Crimson Days usually the week that Valentines day falls on each year).

  • would be a nightmare to implement a single factory to handle all these events!
  • could use the factory method, to implement a lot of factories, but our client code could become incredibly volatile, as we switch out between different factories as events start and end.

Abstract factory pattern

What we need is a way of grouping our factories together, and then accessing them through an interface.

We could then choose initialise an abstract factory through a constructor, or provide a static method that will return an instance of a factory for us depending on the event.
