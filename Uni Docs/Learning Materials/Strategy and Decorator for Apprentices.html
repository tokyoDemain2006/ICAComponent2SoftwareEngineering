“Inheritance is a cool way to change behavior. But we know that it's brittle…There's a tight coupling between the base class and the subclass… Composition has a nicer property. The coupling is reduced by just having some smaller things you plug into something bigger.”

— Erich Gamma

Composition

Composition means building complex behaviourby combining simple objects, rather than inheriting from a chain of parent classes. The key mechanism is delegation;your object holds a reference to another object and calls its methods.

"Favor object composition over class inheritance."

Strategy Pattern

“Define a family of algorithms, encapsulate each one, and make them interchangeable. Strategy lets the algorithm vary independently from clients that use it.”

— Gang of Fouron the intent of the Strategy Pattern

  • You have multiple algorithms that do the same job in different ways
  • You want to avoid switch/if-else chainsthat select between behaviours
  • You need to swap behaviourat runtimewithout modifying the consuming class
  • You want the consuming class to be closed for modificationwhen new behavioursare added

SOLID Principle

How Strategy Implements It

Open/Closed

New strategies are added as new classes; the context class never changes

Dependency Inversion

The context depends on the IStrategy interface, not on any concrete strategy

Single Responsibility

Each strategy class handles one algorithm; the context handles orchestration

Liskov Substitution

Any strategy can replace any other; the context works the same regardless

NotificationService never changes.

No switch. No modification. Just extension.

Decorator Pattern

“Attach additional responsibilities to an object dynamically. Decorators provide a flexible alternative to subclassing for extending functionality.”

— Gang of Fouron the intent of the Decorator Pattern

  • You need to add cross-cutting concerns(logging, caching, retry, auth) to existing objects
  • You want to combine responsibilitiesin any order without a class-per-combination
  • You need to add behaviourwithout modifyingthe original class
  • The responsibilities are independent; logging doesn't depend on retry

SOLID Principle

How Decorator Implements It

Open/Closed

New behaviour is added by wrapping with a new decorator class; the original class never changes

Single Responsibility

Each decorator handles one concern (logging, retry, rate limiting); the core class handles only its core job

Dependency Inversion

Both the decorator and the core class depend on the same interface; neither knows about the other's concrete type

Decorator -Visualised

Decorator - The Problem

You need logging, retry, and rate limiting on your notification channel. Without Decorator:

public class EmailChannel : INotificationChannel

{

// 1. Email logic

// 2. Logging

// 3. Retry (up to _maxRetries)

// 4. Rate limiting (_semaphore)

//

// Four responsibilities in one class.

// SmsChannel needs the same — copy-paste.

}

Four reasons to change:

  1. Email-sending logic
  2. Logging behaviour
  3. Retry policy
  4. Rate limiting

SmsChannel needs the exact same copy-pasted in.

Decorator: one class, one concern.

Reusable across any channel.

"A class should have one, and only one, reason to change." — Uncle Bob!

Decorator - The Solution

public abstract class NotificationChannelDecorator: INotificationChannel

{

protected readonlyINotificationChannel_inner;

protected NotificationChannelDecorator(INotificationChannelinner)

{

_inner = inner;

}

public virtual async Task SendAsync(string recipient, string message)

{

await _inner.SendAsync(recipient, message);

}

}

Decorator - The Solution

public class LoggingDecorator: NotificationChannelDecorator

{

private readonlyILogger<LoggingDecorator> _logger;

public LoggingDecorator(INotificationChannelinner, ILogger<LoggingDecorator> logger)

: base(inner)

{

_logger = logger;

}

public override async Task SendAsync(string recipient, string message)

{

_logger.LogInformation("Sending notification to {Recipient}", recipient);

try

{

await base.SendAsync(recipient, message);

_logger.LogInformation("Notification sent successfully to {Recipient}", recipient);

}

catch (Exception ex)

{

_logger.LogError(ex, "Notification failed for {Recipient}", recipient);

throw; // Let the caller (or another decorator) handle it

}

}

}

public class RetryDecorator: NotificationChannelDecorator

{//ommitted

public override async Task SendAsync(string recipient, string message)

{

for (int attempt = 1; attempt <= _maxRetries; attempt++)

{

try

{

await base.SendAsync(recipient, message);

return; // Success —stop retrying

}

catch (Exception ex) when (attempt < _maxRetries)

{

//catch it

}

}

// Final attempt —let exceptions propagate

await base.SendAsync(recipient, message);

}

}

Decorator - Stacking

Decorators compose — each wraps the previous, like Russian dolls:

// Start with a core strategy

INotificationChannelchannel = new EmailChannel();

// Wrap it with retry —if sending fails, try again

channel = new RetryDecorator(channel, maxRetries: 3, logger);

// Wrap that with logging —log every attempt and result

channel = new LoggingDecorator(channel, logger);
