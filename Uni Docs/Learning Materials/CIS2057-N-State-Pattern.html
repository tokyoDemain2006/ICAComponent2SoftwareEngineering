The State Pattern

The State Pattern

  • The State pattern allows an object to change its behaviour based on its internal state.
    • What does that mean?
  • Returning to our MMORPG, let us pretend that a wizard can be in one of three different emotional / physical states:
    • Rested
    • Focused
    • Fatigued
  • Each state has implications for the character's behaviour.

The State Pattern

  • When a wizard is normal:
    • she is able to fight.
    • she is unable to cast a spell.
  • When a wizard is focused:
    • she is unable to fight.
    • she is able to cast a spell.
    • she does cast a spell, he/she will become fatigued.
  • When a wizard is fatigued:
    • she is able to fight, badly.
    • she is unable to cast a spell.
  • any time:
    • she can meditate to become focused.
    • she can rest to become normal.

Problems with this solution?

  • Too much selection (switch, if/then), which is hard to maintain.
  • Stable code is mixed in with volatile (state-dependent) code.
  • The stable code (Meditate(), Sleep()) applies equally to all object states, and does not change much during the lifespan of the class.
  • The methods Fight() and Cast() are volatile because we are likely to extend the number of states as the game develops.
  • The solution - the state pattern!

Some considerations (1)

  • Stateless versus Stateful States
    • What does this mean?
  • Stateless states could be shared between different context objects
  • Stateful states are unlikely to be shared between context objects – why?
  • Should State objects be created once or as required
    • Problem context specific:
      • Can theresources be spared to have objects created but not currently in use, e.g. embedded devices with limited memory (e.g. Arduinos)?
      • How many states?
      • the states change frequently/infrequently – What cost implications are involved?

Some considerations (2)

  • Stateful may need to be
    • Created as required, to ensure their own internal state is valid, or
    • Late, controllable initialisation/destruction maybe required – common pattern entry and exit methods
      • entry to setup the State object to a suitable initial configuration
      • exit to clean up the State object, particularly if it is managing “costly” resources that need releasing

Summary

  • The State pattern
  • Sometimes we have a class whose overall behaviour changes as it passes through several well-known states.
  • The context object appears to change type/class, when in fact its behaviour is altered by delegating to another object
  • Rather than using selection statements (switch, if/then) to implement these states within the class, we should encapsulate the states and allow them to vary independently of the class.
  • Changes to the state code implementation do affect the context (i.e. the Wizard) code implementation – if the interface remains the same...

Summary of the State pattern

  • Advantages of this pattern:
    • The usual reduced selection structures (if/then, switch).
    • States can vary independently of the context class.
    • You can add states without disturbing the context class.
  • Disadvantages:
    • Object explosion.
    • Where there are only a few states that unlikely to change, this pattern maybe overkill
    • Depending on approach taken, may have objects in memory that are not being used

Circuit-breaker Pattern

Architectural Pattern that uses the State Pattern

Circuit-breaker Pattern

  • Used for handling faults such as unreliable network connection
  • enhanced version of another pattern known as the retry pattern
  • Uses the state pattern to control the retry attempts & variable time between retries

Environmental Sensor anecdote

  • Used a State pattern, collaborating with the Adapter pattern to handle sensor behaviour
  • Sensors could fail, for example they became unseated, and errors would occur
    • effect, similarities to the Circuit-break Pattern, but tailored to needs of the specific sensors
  • The Sensor was the context with typically Normal, Reset, Critical Failure states
    • Normal – Collecting measurements as expected
    • Reset – Attempting to reinitialise/reestablish sensor connectivity
    • Critical – After a predefined number of reset attempts have failed, the sensor is deemed to have critically failed (i.e. dead!)
      • This provided a clean mechanism to make visible/broadcast the failure
