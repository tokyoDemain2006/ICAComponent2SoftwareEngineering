Chain of Responsibility pattern

  • Previously, we covered the Observer pattern.
    • Observers are notified about changes in the state of a subject.
  • Now we move on to another pattern that involves notification.
    • This time the notification (message) is passed to a linear sequence of objects.
  • The first object that is qualified to handle the notification – ‘deals with it’.
  • There is usually some aspect of escalation within this pattern
    • The chain of objects usually forms a hierarchy.
    • real world example: Your student-problems could be escalated from: Module Leader -> Course Leader -> Department Head -> Dean.

Name:

Observer pattern

Type:

Behavioural (communication between objects)

Definition:

Defines a one-to-many dependency between objects so that when one object changes state, all its dependents are notified and updated automatically.

Name:

Chain of Responsibility

Type:

Behavioural

Definition:

Avoid coupling the sender of a request to its receiver by giving more than one object a chance to handle the request.

Chain the receiving objects and pass the request along the chain until an object handles it

How do I use the pattern?

  • start by defining an interface*, that will be implemented by all elements in our chain. Why would we want to do this?
    • Let’s call the interfaceIHandler
  • This interface must define an abstract method which is called when a notification is passed on.
    • will call the method ProcessRequest(intlevel);
  • Notifications have different levels of importance:
    • Low level (1), medium level (2), or high level (3).
  • Let’s assume there are three types of objects in the chain:
    • They are objects of type LowHandler, MediumHandler, and HighHandler respectively.
  • want the notification to be handled by the correct handler.

*Does it have to be an interface?

Chain of Responsibility pattern (part 2.)

  • IHandlerdefines the interface required to handle the request, while theConcreteHandlers handle requests that they are responsible for.
  • the ConcreteHandler cannot handle the request, it passes the request onto its successor, which it maintains a link to.
  • The objects in the chain just need to know how to forward the request to other objects. This decoupling is a huge advantage, as you can change the chain at runtime.

Refinements

  • Implement as an abstract class called Handler
  • Remove instance specific code into an abstract polymorphic method, called Handle().
  • All chain elements extend Handler and implement Handle().
  • Note that interfaces and abstract classes can often be used interchangeably in most design patterns.
    • Matter of taste?
    • because a class already extends from another class.

public abstract class Handler(int level)

{

public Handler? Next { get; set; } = null;

public int Level { get; } = level;

public void ProcessRequest(int requestLevel)

{

if(Level == requestLevel)

{

Handle();

}

else

{

Next?.ProcessRequest(requestLevel);

}

}

protected abstract void Handle();

}

public class LowHandler() : Handler(1)

{

protected override void Handle() => Console.WriteLine("LH: I can handle this request.");

}

public class MediumHandler() : Handler(2)

{

protected override void Handle() => Console.WriteLine("MH: I can handle this request.");

}

public class HighHandler() : Handler(3)

{

protected override void Handle() => Console.WriteLine("HH: I can handle this request.");

}

Further-further-further refinements

  • Jumps in the chain:
    • Use where certain predictable escalations are known.
    • text description contains ‘violence’, escalate to Dean.
  • Branches in the chain:
    • financial problem, route to accounting chain.
    • attendance problem, route to student retention officer.
  • Simplify construction of chain:
    • Always add new chain elements to first element in chain.
    • The new chain element is passed along the chain until it finds the right place (which is usually the end).

Chain of Responsibility - Benefits

  • Decouples the sender of the request from its receivers.
  • Simplifies your object because it doesn’t have to know the chain’s structure and keep direct references to its members.
  • Allows you to add, remove or alter responsibilities dynamically by changing the members or order of the chain.

Applicability

  • Use the Chain of Responsibility Pattern when:
    • More than one object may handle a request, and the handler is not known from first observation.
    • You want to issue a request to one of several objects without specifying the receiver explicitly.
    • The set of objects that can handle a request should be specified dynamically.

Implementation Notes

  • FaultHandler is an abstract class rather than an interface.
  • There are various methods implemented purely for the demonstration testing
    • For example, creating random things!
  • FaultReport can contain many faults, some of which will be the same fault type.
    • The demo just generates a generic fault instance for the various fault types. A refinement would be to have more granular fault messages.
  • FaultReport must be cleared of all faults to be considered successful.
    • Unknown faults will never be removed.

Let’s add a new fault handler …

  1. Enum - Fault.java
    1. Add a new Enum for the new type of fault
    2. Add a new create method for the new Fault enum with a generic fault message
  2. Create a new concrete FaultHandler type to handle this type of fault
  3. Class FaultHanderFactory
    1. Update the FaultHanderFactory to instantiate the new FaultHandler type
  4. Class Bezzler
    1. Add a new FaultHandler reference, the one creating in (3) and update the constructor add it to the chain
